# Process Description

## Repository Structure

### Initial Setup

The Git repository for a new project must be created with the following structure:

- **Sub-repository Inclusion**: The current project (Agent Based Software Engineering Process assets) must be included as a sub-repository within the new project repository. This ensures that all methodology documentation is readily available and can be referenced throughout the project lifecycle.

- **Documentation Folder**: The new project repository must include a dedicated documentation folder. This folder serves multiple critical purposes:
  - **Input Documentation**: Contains all documents required as input for this methodology
  - **Accumulated Documentation**: Stores all documentation generated by the process throughout the project lifecycle
  - **AI Agent Context Source**: Serves as the primary context source for AI agents, providing them with the necessary information to understand project history, requirements, and implementation decisions

## Change Management Structure

### Change Components

All code changes, including initial versions, must consist of three essential components:

1. **Change Requirement**
   - A clear, documented statement of what needs to be changed or implemented
   - Includes the rationale, objectives, and expected outcomes of the change

2. **Implementation Plan**
   - A structured plan detailing how the change will be implemented
   - Specifies the approach, affected components, and adherence to relevant guidelines
   - Addresses both functional and non-functional considerations as applicable

3. **Code Change**
   - The actual code modifications, additions, or deletions resulting from the implementation plan
   - Must align with the documented Implementation Plan

### Documentation Retention

All Change Requirements and Implementation Plans must be preserved as part of the AI Agents context documentation. This ensures:

- **Historical Context**: AI agents can understand the evolution of the codebase and the reasoning behind past decisions
- **Traceability**: Changes can be traced back to their original requirements and implementation rationale
- **Consistency**: Future changes can reference and build upon previous implementations
- **Impact Analysis**: Agents can assess how new changes relate to existing implementations

## Change Request Process

### Analysis Phase

Whenever a change is requested, the AI agent must perform a comprehensive analysis that includes:

1. **Context Review**
   - Analyze the accumulated documentation from previous changes
   - Review Change Requirements and Implementation Plans from the project history
   - Understand the current state of the project's requirements and architecture

2. **Codebase Assessment**
   - Examine the actual codebase to understand current implementation
   - Identify existing components, patterns, and architectural decisions
   - Assess the current state of code organization and structure

3. **Impact Assessment**
   - Determine the scope and impact of the requested change
   - Identify which components, modules, or systems will be affected
   - Evaluate potential side effects and dependencies

### Change Classification

The AI agent must classify the change by identifying:

- **Functional Requirements Change**: Changes that affect what the system does, including:
  - New features or capabilities
  - Modifications to existing functionality
  - Removal of features
  - Changes to business logic or user interactions
  
  An assessment of the impact of the change must be performed as stated in the Change Request Assessment Description document (2_1_change_request_assessment_description.md) in the Functional Requirements Handling and Usage folder.

- **Non-Functional Requirements Change**: Changes that affect how the system performs, including:
  - Performance optimizations
  - Security enhancements
  - Scalability improvements
  - Maintainability and code quality improvements
  - Reliability and availability enhancements
  
  An assessment of the impact of the change must be performed as stated in the Change Request Assessment Description document in the Non-Functional Requirements Handling and Usage folder.

A single change may involve both functional and non-functional aspects, and both must be identified and addressed.

### Guideline Identification

Based on the change classification and impact assessment, the AI agent must:

- Identify the relevant guidelines from the methodology documentation that apply to the change
- Reference specific sections of the documentation that provide guidance for:
  - Functional Requirements Handling and Usage (for functional changes)
  - Non-Functional Requirements Handling and Usage (for non-functional changes)
  - Change Management and Implementation (for all changes)

### Implementation Plan Generation

The AI agent must generate an Implementation Plan that:

- Addresses all identified guidelines from the relevant methodology sections
- Specifies the approach for implementing the change
- Identifies the components that will be modified, added, or removed
- Ensures adherence to established patterns and practices
- Minimizes code entropy by limiting the scope of changes to necessary components
- Maintains consistency with previous implementations documented in the project history

The Implementation Plan serves as a blueprint that guides the code change implementation and ensures that all methodology guidelines are properly addressed.
